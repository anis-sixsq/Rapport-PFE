\section*{Introduction} 
Ce chapitre contient une description complète du comportement du système à développer. Ce dernier présente une analyse et une spécification des différents besoins fonctionnels et non fonctionnels. Nous identifions, dans une première partie, les acteurs du système. Dans une deuxième partie, nous itérons les exigences fonctionnelles et non fonctionnelles. Ensuite, nous analysons les besoins à travers l'élaboration des diagrammes de cas d'utilisation et de séquence système qui décrivent toutes les interactions possibles entre les utilisateurs et la solution en termes de fonctionnalités.




\section{Identification des acteurs du système}
Les acteurs évoqués dans le système sont :

\begin{itemize}
\item  L'utilisateur de Hadoop :
\begin{itemize}
\item Est un simple utilisateur qui a un compte SlipStream.
\item Il doit être un client au moins d'un seul fournisseur Cloud (IaaS).
\item Il a le droit de déployer notre solution Hadoop dans une infrastructure Cloud.
\item Il a le droit d'utiliser et gérer la solution Hadoop déployée.
\item Il a le droit de terminer la solution Hadoop déployée.
\end{itemize} 


\item L'administrateur de la solution Hadoop :
\begin{itemize}
\item Est un utilisateur SlipStream qui a le droit d'éditer la solution.
\item Il a le droit de voir et modifier la configuration des VMs de la solution.
\item II a le droit de mettre à jour la solution.
\end{itemize} 
\end{itemize} 





\section{Spécification des besoins}
La spécification des besoins met en relief les fonctionnalités utiles que doit fournir le système. La solution à réaliser doit satisfaire les besoins fonctionnels et non fonctionnels énumérés dans les deux prochaines sections.




\subsection{Besoins fonctionnels}
Nous détaillons dans cette partie les principales fonctionnalités, que le système doit fournir aux différents acteurs, qui se présentent comme suit:

\begin{itemize}
\item  \textbf{Utilisateur de Hadoop} La solution doit permettre aux utilisateurs:
\begin{itemize}
\item \textbf{Le déploiement d'un cluster Hadoop} : le système doit permettre aux utilisateurs de déployer, en un seul clic, un cluster Hadoop dans un ou plusieurs Clouds au choix.
\item \textbf{L'utilisation des machines de Hadoop} : une fois Hadoop déployé, le système doit permettre aux utilisateurs de connecter à la machine «master» en mode SSH ou en mode graphique et utiliser tous les fonctionnalités de Hadoop « 2.x ».
\item \textbf{La gestion des services de Hadoop} : une fois Hadoop déployé, le système doit permettre aux utilisateurs de gérer les différents services de Hadoop.
\item \textbf{La gestion du cluster Hadoop} : une fois Hadoop déployé, le système doit permettre aux utilisateurs d'ajouter ou de terminer des machines «slaves» du cluster selon le besoin.
\item \textbf{La gestion des VMs du cluster} : une fois Hadoop déployé, le système doit permettre aux utilisateurs de gérer les caractéristiques des machines virtuelles utilisées dans le déploiement.
\end{itemize} 



\item \textbf{Administrateur de la solution Hadoop}  La solution doit permettre à l'administrateur:
\begin{itemize}
\item \textbf{La mise à jour de la solution} : le système doit permettre à l'administrateur de mettre à jour la version de Hadoop et les autres outils utilisés dans la solution.
\item \textbf{La gestion des machines préconfigurées de Hadoop} : le système doit permettre à l'administrateur de gérer les types d'instances et les images ID des VMs du cluster.
\item \textbf{La gestion de la sécurité au niveau de Cloud} : le système doit permettre à l'administrateur de gérer le groupe de sécurité.\\
\end{itemize} 
\end{itemize} 


\textbf{Remarque}:\\
Dans la suite de ce chapitre, nous ne détaillerons que les cas d'utilisation de l'acteur « Utilisateur de Hadoop » puisque tout le travail à réaliser est lié directement à cet acteur. Pour l'acteur « administrateur », c'est la plateforme SlipStream qui gère tous ces cas d'utilisation.\\





\subsection{Besoins non fonctionnels}
Notre objectif dans ce projet est de développer une solution performante. Etant donné qu'une application uniquement fonctionnelle et opérationnelle ne garantit ni la satisfaction ni la fidélité des utilisateurs, nous devons prendre en considération des critères non fonctionnels lors de la conception et l'implémentation de notre solution. Parmi ces exigences, nous citons:

\begin{itemize}
\item \textbf{Le temps de déploiement} : la solution doit permettre à l'utilisateur de déployer un cluster Hadoop dans un temps très réduit par rapport un déploiement manuel.
\item \textbf{L'ergonomie} : pendant le temps de déploiement, la solution doit afficher à l'utilisateur toutes les informations qui concernent l'état de déploiement. Après la fin de déploiement, la solution doit fournir à l'utilisateur toutes les informations nécessaires pour utiliser le cluster Hadoop déployé.
\item \textbf{La performance des machines} : la solution doit permettre à l'utilisateur de déployer un cluster Hadoop performant qui répond à ses besoins en puissance de calcul.
\item \textbf{La sécurité} : l'application doit respecter certaines règles relatives à la sécurité des systèmes informatiques. En effet, la solution génère d'une manière aléatoire des mots de passe complexes pour permettre à l'utilisateur de s'authentifier au niveau des VMs et les interfaces d'administration.
\item \textbf{La maintenance} : les différentes parties de la solution doivent être faciles à maintenir. Par conséquent, le code doit être lisible, bien commenté et bien structuré.
\end{itemize}
